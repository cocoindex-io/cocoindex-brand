Build your first flow in 10 minutes.

This guide walks you through a minimal cocoindex flow that indexes a folder of markdown files into a vector store — the "hello world" every framework should earn first.

Time~10 minutes
LanguagePython 3.10+
RequiresPostgres 14+
Versionv 0.9.4
Last updatedApr 14, 2026
i
New to the mental model? CocoIndex follows dataflow programming — each transform creates a new field from input fields, no hidden state, full lineage out of the box. If that's unfamiliar, skim Indexing basics first. Otherwise, keep going.

Install CocoIndex & Postgres.

CocoIndex ships as a Python package backed by a native Rust engine. It uses Postgres as its internal state store for incremental processing — install it first if you don't already have one running.

# install the package
pip install -U cocoindex

# verify
cocoindex --version
# → cocoindex 0.9.4

Need a local Postgres? The fastest route:

# spin up a local postgres with pgvector
docker run -d --name coco-pg \
  -e POSTGRES_PASSWORD=cocoindex \
  -p 5432:5432 \
  pgvector/pgvector:pg16

Scaffold a new project.

Create a directory and drop in a couple of markdown files to play with. In production you'd point at a real folder, a bucket, or a database.

mkdir coco-quickstart && cd coco-quickstart
mkdir markdown_files
echo "# Hello CocoIndex\n\nA small coconut, with a face." > markdown_files/hello.md
touch main.py .env

Open .env and point CocoIndex at your Postgres instance:

COCOINDEX_DATABASE_URL="postgres://postgres:cocoindex@localhost:5432/postgres"

Declare the flow.

A flow is a plain Python function decorated with @cocoindex.flow_def. Inside, you declare sources, transforms, and a collector that writes to a target. CocoIndex takes care of scheduling, caching, schema evolution, and incremental updates.

import cocoindex

@cocoindex.flow_def(name="TextEmbedding")
def text_embedding_flow(
    flow_builder: cocoindex.FlowBuilder,
    data_scope: cocoindex.DataScope,
):
    # 1) read files from disk
    data_scope["documents"] = flow_builder.add_source(
        cocoindex.sources.LocalFile(path="markdown_files")
    )

    # 2) prepare a collector for the target
    doc_embeddings = data_scope.add_collector()

    # 3) transform each document row
    with data_scope["documents"].row() as doc:
        doc["chunks"] = doc["content"].transform(
            cocoindex.functions.SplitRecursively(),
            language="markdown",
            chunk_size=2000,
            chunk_overlap=500,
        )

        with doc["chunks"].row() as chunk:
            chunk["embedding"] = chunk["text"].transform(
                cocoindex.functions.SentenceTransformerEmbed(
                    model="sentence-transformers/all-MiniLM-L6-v2",
                )
            )
            doc_embeddings.collect(
                filename=doc["filename"],
                location=chunk["location"],
                text=chunk["text"],
                embedding=chunk["embedding"],
            )

    # 4) write to Postgres with a vector index
    doc_embeddings.export(
        "doc_embeddings",
        cocoindex.storages.Postgres(),
        primary_key_fields=["filename", "location"],
        vector_indexes=[
            cocoindex.VectorIndex(
                "embedding",
                cocoindex.VectorSimilarityMetric.COSINE_SIMILARITY,
            ),
        ],
    )

What just happened?

add_source()
Declares where data comes from. Produces a typed table — here, filename and content fields.
.row() context
"For each row of this table." Transforms inside the with block run per-row, and new fields land on that row.
SplitRecursively
Built-in chunker. Understands markdown, code (Tree-sitter), prose. Produces a nested chunks table with location and text.
.collect()
Pulls selected fields into the collector. Multiple collect calls merge into a single output table.
.export()
Declares the target. CocoIndex reconciles: creates tables, adds indexes, keeps it in sync on every run.

Run the flow.

Two commands. The first sets up the internal storage and target tables. The second runs the flow — and will keep it running, watching for changes.

cocoindex setup main.py
cocoindex update main.py --live
●
What to expect Your first run indexes everything. Subsequent runs only process what changed — a new file, a changed chunk, an updated embedder. This is the core promise: only the delta, at any scale.

Open CocoInsight.

CocoInsight is a local, browser-based view of your running flow. It renders lineage, shows per-step counts, and lets you click through to the exact rows that changed. Think of it as dev tools, for data.

cocoindex server main.py --insight
# → open http://localhost:8765
Flow · TextEmbedding● live · 94% cache hit

Query the index.

Anything that speaks Postgres + pgvector can query it — including your favorite ORM. Here's a tiny search helper for a quick smoke test.

import cocoindex, psycopg

@cocoindex.transform_flow()
def embed_query(text: cocoindex.DataSlice[str]):
    return text.transform(
        cocoindex.functions.SentenceTransformerEmbed(
            model="sentence-transformers/all-MiniLM-L6-v2"
        )
    )

def search(q: str, k: int = 5):
    v = embed_query.eval(q)
    with psycopg.connect(DB_URL) as c:
        return c.execute(
            """SELECT filename, location, text,
                      1 - (embedding <=> %s::vector) AS score
                 FROM doc_embeddings
                 ORDER BY embedding <=> %s::vector
                 LIMIT %s""",
            (v, v, k),
        ).fetchall()

Where to go next.

You have a live, incremental, explainable index. Every file change propagates, and only the delta is recomputed. From here:

  • Swap in another source — point at Google Drive, a Postgres table, or an S3 bucket.
  • Extract structure with an LLM — entities, decisions, action items from free-form text.
  • Build a graph target — same flow, Neo4j or Kuzu on the other end.
  • Deploy to production — the control plane is the same whether it's your laptop or 10⁹ rows.
!
Gotcha: model consistency Use the same embedding model for indexing and querying. The @cocoindex.transform_flow() decorator exists precisely to share that transform across both paths — don't instantiate a second embedder by hand.
