01 · Part one

Setup

Install CocoIndex and the LLM client. Point at a Postgres and drop in your LLM provider key.

01
Part

Setup once · ~3 min

CocoIndex + aiohttp for async HTTP + LiteLLM for the LLM calls. Any Postgres works — Docker if you don't have one.

Steps 2 Files 0
1.1 Install · 30 s

Install the five dependencies.

CocoIndex brings the engine. LiteLLM normalizes the LLM API across providers (OpenAI, Gemini, Anthropic). aiohttp handles async HTTP to HN.

  • cocoindex, asyncpg, aiohttp, litellm, pydantic
  • Python 3.10+ for str | None unions in the models.
bash shell 
uv init hn-trending && cd hn-trending
uv add cocoindex asyncpg aiohttp litellm pydantic
1.2 Env · 1 min

Postgres + LLM key.

CocoIndex reads POSTGRES_URL from the env. LiteLLM picks up the provider key that matches your model string.

  • Default model: gemini/gemini-2.5-flash → set GEMINI_API_KEY.
  • Swap for openai/gpt-4o-mini → set OPENAI_API_KEY.
bash shell 
docker run -d --name coco-pg \
  -e POSTGRES_USER=cocoindex \
  -e POSTGRES_PASSWORD=cocoindex \
  -p 5432:5432 postgres:16

export POSTGRES_URL="postgres://cocoindex:cocoindex@localhost/cocoindex"
export GEMINI_API_KEY="..."   # or OPENAI_API_KEY / ANTHROPIC_API_KEY
Tip

No vector extension needed here — this pipeline uses plain SQL, not pgvector. Any managed Postgres is fine.

02 · Part two

Model the data.

Two shapes to write down: what comes out of HN, and what you want in Postgres. Everything else is derived.

02
Part

Types first. Schema is derived.

Domain types for HN threads and comments on one side; flat row dataclasses for the two Postgres tables on the other.

Steps 2 File main.py
2.1 Domain · 2 min

Thread in, comment tree attached.

HN returns nested data. We flatten the comment tree into a list hanging off each thread — then every node becomes one row downstream.

  • Dataclasses, not Pydantic — no runtime overhead on the hot path.
  • created_at is a real datetime so SQL ordering works.
main.py — HN domain python 
from dataclasses import dataclass
from datetime import datetime

@dataclass
class Comment:
    id: str
    author: str | None
    text: str | None
    created_at: datetime | None

@dataclass
class Thread:
    id: str
    author: str | None
    text: str                      # title + body
    url: str | None
    created_at: datetime | None
    comments: list[Comment]
2.2 Target tables · 2 min

Two rows. Two tables.

Thread and comment both land in hn_messages — content_type tells them apart. Topics get their own join-friendly table keyed on (topic, message_id).

  • Composite PK on hn_topics → same topic can mention many messages.
  • One TableTargets container lets both tables travel as a unit.
main.py — target tables python 
@dataclass
class HnMessage:
    id: str; thread_id: str
    content_type: str             # 'thread' | 'comment'
    author: str | None
    text: str | None
    url: str | None
    created_at: datetime | None

@dataclass
class HnTopic:
    topic: str
    message_id: str
    thread_id: str
    content_type: str
    created_at: datetime | None

@dataclass
class TableTargets:
    messages: postgres.TableTarget[HnMessage]
    topics:   postgres.TableTarget[HnTopic]
03 · Part three

Fetch from HackerNews.

Two thin async helpers hit the HN Algolia API — one for the recent thread list, one for the full tree. This is where a custom source becomes worth it.

03
Part

Two endpoints. One typed result.

List the thread IDs, then hydrate each with a recursive walk of the comment tree. Keep the aiohttp.ClientSession local — each fan-out component gets its own.

Steps 2 File main.py
3.1 List · 1 min

Ask Algolia for recent stories.

A single search_by_date call returns up to max_results story IDs, sorted freshest first. No auth needed.

  • Keep this tiny — just the IDs. The tree comes later.
  • Change tags to ask_hn, show_hn, etc. for narrower firehoses.
main.py — fetch_thread_list python 
async def fetch_thread_list(session, max_results=MAX_THREADS):
    url = "https://hn.algolia.com/api/v1/search_by_date"
    params = {"tags": "story", "hitsPerPage": max_results}
    async with session.get(url, params=params) as resp:
        resp.raise_for_status()
        data = await resp.json()
        return [h["objectID"] for h in data.get("hits", [])]
3.2 Hydrate · 2 min

Walk the comment tree recursively.

One /items/{id} call returns the whole thread with arbitrarily nested children. We flatten them into a single list so each becomes a row.

  • Combine title and text into one thread.text — both feed the LLM.
  • Parse ISO timestamps at ingest — never in SQL.
main.py — fetch_thread python 
async def fetch_thread(session, thread_id):
    url = f"https://hn.algolia.com/api/v1/items/{thread_id}"
    async with session.get(url) as resp:
        data = await resp.json()

    comments: list[Comment] = []
    def walk(node):
        for child in node.get("children", []):
            if (cid := child.get("id")):
                ctime = child.get("created_at")
                comments.append(Comment(
                    id=str(cid), author=child.get("author"),
                    text=child.get("text"),
                    created_at=datetime.fromisoformat(ctime) if ctime else None,
                ))
            walk(child)
    walk(data)

    text = data.get("title", "")
    if body := data.get("text"): text += "\n\n" + body
    return Thread(id=thread_id, author=data.get("author"),
                  text=text, url=data.get("url"),
                  created_at=datetime.fromisoformat(data["created_at"]),
                  comments=comments)
Why custom, not built-in?

There's no first-party HN source — so we write ~30 lines. The upside: the exact same @coco.fn + declare_row machinery below works unchanged.

04 · Part four

Extract topics with an LLM.

Loose prose in, a typed list of topic strings out. The Pydantic schema is the contract — LiteLLM enforces it on the model's response.

04
Part

Prompt once. Validate always.

A field description doubles as the prompt instructions. The response has to be a TopicsResponse — the model can't return prose.

Steps 2 Files models.py · main.py
4.1 Schema · 2 min

The schema is the prompt.

LiteLLM passes the Field(description=...) straight to the model as part of the structured-output spec. Write your rules here — casing, acronyms, normalization.

  • "Claude" — product name · "Anthropic" — company · "React" — library.
  • Phrases get normalized into multiple canonical topics.
models.py python 
from pydantic import BaseModel, Field

class TopicsResponse(BaseModel):
    """Response containing a list of extracted topics."""

    topics: list[str] = Field(
        description="""List of extracted topics.

Each topic can be a product, technology, model, person,
company, or business domain.
Capitalize for proper nouns and acronyms only.
Avoid acronyms unless popular and unambiguous.

Examples:
  - "Anthropic" (not "ANTHR")
  - "Claude"   (product name)
  - "React"    (well-known library)
  - "PostgreSQL"

Normalize phrases into multiple topics when useful.
For people, prefer first + last name.
"""
    )
4.2 Extractor · 2 min

One @coco.fn does the work.

Decorating with @coco.fn makes it a first-class pipeline step — cacheable, observable, retried. Inside, it's a normal async call to LiteLLM.

  • Truncate to 4k chars — HN comments can get long.
  • Return empty list for empty input so the engine doesn't waste calls.
main.py — extract_topics python 
from litellm import acompletion
from models import TopicsResponse

LLM_MODEL = "gemini/gemini-2.5-flash"

@coco.fn
async def extract_topics(text: str | None) -> list[str]:
    if not text or not text.strip():
        return []

    resp = await acompletion(
        model=LLM_MODEL,
        messages=[{"role": "user",
                   "content": f"Extract topics from:\n\n{text[:4000]}"}],
        response_format=TopicsResponse,
    )
    content = resp.choices[0].message.content
    return TopicsResponse.model_validate_json(content).topics
Swap providers in one line

Change LLM_MODEL to openai/gpt-4o-mini or anthropic/claude-haiku-4-5. Same function, different API key, same structured response.

05 · Part five

Declare the pipeline.

Two functions tie everything together. Each thread fans out into one message + many comments + topic rows. CocoIndex diffs the declarations and writes only the delta.

05
Part

Fan out. Declare. Done.

process_thread owns the work for one thread; app_main fans out over the list. No sync code.

Steps 2 File main.py
5.1 process_thread · 3 min

One thread → many rows.

Fetch the thread, extract thread-level topics, declare the thread row, then loop through comments doing the same. Each declare_row is a statement of "this should exist."

  • Same thread_id links comments back to their parent for joins.
  • If a thread disappears next sync, all its messages + topics clean up automatically.
main.py — process_thread python 
@coco.fn
async def process_thread(thread_id: str, targets: TableTargets):
    async with aiohttp.ClientSession() as session:
        thread = await fetch_thread(session, thread_id)
    thread_topics = await extract_topics(thread.text)

    targets.messages.declare_row(row=HnMessage(
        id=thread.id, thread_id=thread.id, content_type="thread",
        author=thread.author, text=thread.text, url=thread.url,
        created_at=thread.created_at,
    ))
    for topic in thread_topics:
        targets.topics.declare_row(row=HnTopic(
            topic=topic, message_id=thread.id, thread_id=thread.id,
            content_type="thread", created_at=thread.created_at,
        ))

    for c in thread.comments:
        topics = await extract_topics(c.text)
        targets.messages.declare_row(row=HnMessage(
            id=c.id, thread_id=thread.id, content_type="comment",
            author=c.author, text=c.text, url="", created_at=c.created_at,
        ))
        for t in topics:
            targets.topics.declare_row(row=HnTopic(
                topic=t, message_id=c.id, thread_id=thread.id,
                content_type="comment", created_at=c.created_at,
            ))
5.2 app_main · 2 min

Mount tables, fan out over threads.

Create both table targets with mount_table_target. Fetch the thread ID list. Kick off process_thread for each via mount_each — parallel, change-detected, memoizable.

  • pg_schema_name="coco_examples" keeps your app isolated.
  • coco.mount_each takes (key, args) tuples for stable component paths.
main.py — app_main python 
@coco.fn
async def app_main() -> None:
    messages_tbl = await postgres.mount_table_target(
        PG_DB, table_name="hn_messages",
        table_schema=await postgres.TableSchema.from_class(
            HnMessage, primary_key=["id"]),
        pg_schema_name="coco_examples",
    )
    topics_tbl = await postgres.mount_table_target(
        PG_DB, table_name="hn_topics",
        table_schema=await postgres.TableSchema.from_class(
            HnTopic, primary_key=["topic", "message_id"]),
        pg_schema_name="coco_examples",
    )
    targets = TableTargets(messages=messages_tbl, topics=topics_tbl)

    async with aiohttp.ClientSession() as session:
        thread_ids = await fetch_thread_list(session)

    await coco.mount_each(process_thread,
        ((tid, tid) for tid in thread_ids), targets)

app = coco.App(coco.AppConfig(name="HNTrendingTopics"), app_main)

Three ways to run it.

One-shot for CI, live mode for continuous sync, CocoInsight for an interactive DAG view with inspectable rows.

6.2 Query · 1 min

Trending in one SQL statement.

Weight threads higher than comments. Group by topic. Order by score then recency. That's it.

  • THREAD_LEVEL_MENTION_SCORE = 5, comment = 1.
  • For search, join hn_topics back to hn_messages on message_id.
main.py — get_trending_topics python · sql 
async def get_trending_topics(pool, limit=20):
    async with pool.acquire() as conn:
        rows = await conn.fetch("""
            SELECT
                topic,
                SUM(CASE
                    WHEN content_type = 'thread' THEN 5
                    ELSE 1
                END)                        AS score,
                MAX(created_at)             AS latest_mention,
                COUNT(DISTINCT thread_id)   AS thread_count
            FROM hn_topics
            GROUP BY topic
            ORDER BY score DESC, latest_mention DESC
            LIMIT $1
        """, limit)
    return [dict(r) for r in rows]
